# Ariake Events: プロジェクト全容ドキュメント

> **このドキュメントの目的**
> このプロジェクトに初めて参加する開発者（人間・AIを問わず）が、他のどのドキュメントよりも先に読み、**記憶ゼロの状態から最短時間でプロジェクトの全体像をキャッチアップする**ことを目的としています。

## 1. このプロジェクトは何か？ (What & Why)

**Ariake Events** は、東京・有明エリアで開催されるイベント情報を一元的に集約・表示するWebアプリケーションです。

### 背景と目的

有明エリアには、東京ビッグサイト、有明アリーナ、東京ガーデンシアターなど、大規模イベント施設が集中しています。しかし、各施設のイベント情報はそれぞれの公式サイトで個別に発表されるため、利用者は複数のサイトを巡回する必要があり、非常に不便です。

特に、複数の大規模イベントが同日に開催されると、交通機関や周辺施設が極度に混雑し、移動や食事に大きな支障が出ます。この問題を解決するため、本プロジェクトは以下の2点を提供することを目的としています。

1.  **情報の集約**: 有明エリアのイベント情報を1つのサイトで網羅的に閲覧できる環境を提供する。
2.  **混雑の可視化**: イベントの規模や時間帯から混雑リスクを算出し、利用者が事前に混雑を回避できるようにする。

## 2. プロジェクトの現在地 (2026年3月5日時点)

本プロジェクトは、明確に定義された3つのフェーズで構成されています。

| フェーズ | 内容 | 状態 | 担当 |
|:---|:---|:---|:---|
| **フェーズ1** | **データ収集基盤の構築** | ✅ **完了** | **AI (Manus)** |
| フェーズ2 | Webフロントエンドの実装 | ⏳ 未着手 | 人間 / AI |
| フェーズ3 | パーソナライズ機能・高度な混雑予測 | 計画中 | 人間 / AI |

現在は **フェーズ1が完了** し、いつでもフェーズ2に着手できる状態です。

### 完了済みの成果物 (フェーズ1)

-   **イベント情報の自動収集**: 5つの主要施設から毎日定時にイベント情報を自動収集するスクレイパー（TypeScript + Playwright）。
-   **データの一元管理**: 収集したデータを正規化し、`packages/web/public/events.json` としてリポジトリに自動コミットする仕組み。
-   **自動実行環境**: 上記プロセスを毎日 JST 09:00 に自動実行するGitHub Actionsワークフロー。

## 3. 次のステップ：フェーズ2に着手する

次にこのプロジェクトを引き継ぐ方は、**フェーズ2：Webフロントエンドの実装** を担当します。

### 最初のタスク

まず、`packages/web` ディレクトリに、**React (Vite + TypeScript) と Tailwind CSS** を導入してください。これがフロントエンド開発の技術基盤となります。

```bash
# 1. webディレクトリに移動
cd packages/web

# 2. Vite (React + TS) プロジェクトを初期化
pnpm create vite . --template react-ts

# 3. 依存関係をインストール
pnpm install

# 4. Tailwind CSSを導入
pnpm install -D tailwindcss postcss autoprefixer
pnpm exec tailwindcss init -p
```

### 実装すべき機能

開発の具体的な要件は `CONTRIBUTING.md` に記載されています。必ずそちらを参照してください。

> **参照先**: [CONTRIBUTING.md#4-次のステップフェーズ2](https://github.com/KANNOHI1/ariake-events/blob/main/CONTRIBUTING.md#4-%E6%AC%A1%E3%81%AE%E3%82%B9%E3%83%86%E3%83%83%E3%83%97%E3%83%95%E3%82%A7%E3%83%BC%E3%82%BA2)

## 4. 技術的な意思決定の記録 (Decision Log)

プロジェクトを進める上で行われた主要な技術的判断とその理由を記録します。

| 日付 | 決定事項 | 理由 |
|:---|:---|:---|
| 2026-02-20 | **対象施設から「有明コロシアム」を除外** | 公式サイトにテニス等のスポーツ大会情報しか掲載されず、民間イベント（ライブ、展示会など）の情報がほぼ皆無。情報の網羅性を担保できないため、費用対効果が低いと判断。 |
| 2026-02-25 | **データ更新方式を「毎日全件洗い替え」に決定** | 差分更新は実装が複雑になる一方、イベント情報は多くても数百件程度。全件を毎日再生成する方が、シンプルで堅牢な設計になると判断。 |
| 2026-03-05 | **GitHub Actionsのファイル名を `scrape.yml` に統一** | 当初 `scraper.yml` と `scrape.yml` が混在していたが、シンプルさを優先し、動詞である `scrape.yml` に統一。 |
