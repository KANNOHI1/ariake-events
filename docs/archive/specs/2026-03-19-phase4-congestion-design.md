## ゴール

有明エリアの「その日どれくらい混むか」を数値スコアで可視化し、イベントカードとカレンダービューに表示する。

**対象ユーザー:**
- イベント参加者: どの時間帯に行けば混雑を避けられるか
- 周辺居住者: その日の有明エリアの混雑予測を事前に把握する

---

## congestionRisk の定義

**「その日の有明エリア全体への影響度」**を `0.0〜1.0` のスコアで表す。

- スコアは**日単位**で計算する（イベント単位ではなく）
- 各イベントカードには、そのイベントが開催される日のエリアスコアを表示する
- 型定義は既存の `congestionRisk: number | null` をそのまま使用（スキーマ変更なし）
- イベントのない日のスコアは `0.0`（スコア計算から除外するのではなく、明示的に0を設定）

---

## 計算アルゴリズム

### 概念式

```
日別スコア = normalize( Σ 施設スコア[i] )  // 0.0〜1.0 に正規化

施設スコア[i] =
  capacityScore[i]           // 施設キャパ ÷ エリア合計キャパ（イベントがある施設のみ加算）
  × categoryMultiplier[i]    // カテゴリ別係数
  × timePatternFactor[i]     // 来場パターン係数
  × dayTypeBonus[i]          // 曜日・祝日係数（土日祝のみ適用）
```

**注意**: `capacityScore` は「イベントが存在する施設のみ」計算して加算する。イベントのない施設はスコア0として扱い、加算しない。

### 正規化

```typescript
const MAX_POSSIBLE_SCORE = 0.74; // 全施設同日開催・最大係数の合計（計算値: 0.7399）
const normalizedScore = Math.min(rawScore / MAX_POSSIBLE_SCORE, 1.0);
```

> MAX_POSSIBLE_SCORE の内訳:
> - 有明アリーナ (music, 土日): (15000/86000) × 1.0 × 1.0 × 1.05 = 0.183
> - TOYOTA ARENA TOKYO (music, 土日): (10000/86000) × 1.0 × 1.0 × 1.05 = 0.122
> - 東京ガーデンシアター (music, 土日): (8000/86000) × 1.0 × 1.0 × 1.05 = 0.098
> - 東京ビッグサイト (exhibition, N=1, 土日): (50000/86000) × 0.6 × 0.7 × 1.2 = 0.293
> - 有明ガーデン (music, 土日): (3000/86000) × 1.0 × 0.7 × 1.8 = 0.044
> - 合計: 0.740

## カテゴリ係数 `categoryMultiplier`

来場者の時間・空間集中度を反映する。未知カテゴリが追加された場合は `0.6`（`other` 相当）をデフォルトとして適用する。

| カテゴリ | 係数 | 理由 |
|---|---|---|
| `"music"` | 1.0 | 最も集中度が高い（開演前後に集中） |
| `"sports"` | 0.9 | 音楽に次いで高い |
| `"anime"` | 0.8 | ファン層が集中しやすい |
| `"exhibition"` | 0.6 | 開館〜閉館に分散 |
| `"other"` | 0.6 | デフォルト値 |
| `"kids"` | 0.5 | 滞在時間が長く分散 |
| `"fashion"` | 0.5 | 分散型 |
| `"food"` | 0.4 | 最も分散型 |

## 来場パターン係数 `timePatternFactor`

### 施設タイプ別パターン

| 施設タイプ | パターン | 係数の基本値 |
|---|---|---|
| アリーナ系3施設 | 時間集中型 | 1.0 |
| 東京ビッグサイト・有明ガーデン | 分散型 | 0.7 ÷ N |

### 複数日イベントの処理

`N = endDate - startDate + 1`（日数、整数）

- 単日イベント（N=1）: 係数 = 0.7
- 複数日イベント（N>1）: 係数 = max(0.7 ÷ N, 0.3)（最小値0.3を下限とする）
- **中間日の扱い**: `startDate` から `endDate` の各日すべてに同じ係数（0.7 ÷ N）を適用する。期間中の全日が同等の影響を持つとみなす。

> 例: 3/20〜3/22（3日間）の展示会は、3/20, 3/21, 3/22 それぞれの日別スコアに係数 0.7/3 ≈ 0.233 を適用する。

---

## 曜日・祝日係数 `dayTypeBonus`

土日祝の場合にのみ適用する。平日は係数 1.0（乗算なし）。

**根拠**: 住友不動産商業マネジメントのIoT人流実証実験（有明ガーデン内キッズスペース、週末/平日比 2.0倍）および郊外型商業施設の実来客カウント平均（1.24倍、Vieureka調査）を参考に施設別設定。チケット制イベントは曜日に関わらず満員前提のため係数差を最小限にしている。

| 施設 | 平日 | 土日祝 |
|---|---|---|
| 有明ガーデン | 1.0 | **1.8** |
| 東京ビッグサイト | 1.0 | **1.2** |
| アリーナ系3施設 | 1.0 | **1.05** |

### 祝日判定

`date-holidays` パッケージ（`jp` ロケール）を使用して日本の祝日を判定する。スクレイパー実行時に `startDate` の曜日・祝日フラグを計算してスコアに反映する。

## 設計上の確定判断

| 決定事項 | 内容 | 理由 |
|---|---|---|
| スコアの単位 | 日単位（イベント単位ではない） | 居住者・来訪者が知りたいのは「その日の有明全体」 |
| スコアの型 | `number 0.0〜1.0` | 既存型定義と一致、UI表示の柔軟性確保 |
| 曜日係数・有明ガーデン | ×1.8 | IoTデータ（×2.0）を保守的に調整 |
| アーティスト集客力API | Phase 5送り | 実装コスト高・API依存・マッチング精度問題 |
| dayTypeBonus省略なし | 施設別に採用 | 居住者体感 + 実データで裏付けあり |
| イベントのない日 | スコア 0.0 を明示設定 | null と 0.0 を区別しない（常に計算値を格納） |
| 複数日イベントの中間日 | 各日に均等係数を適用 | 期間中の全日を同等影響とみなす |
| 祝日判定 | `date-holidays` パッケージ（jp） | 保守コスト最小、信頼性高い |
